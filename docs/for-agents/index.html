Builder lane, still downstream of the control room

Give local coding hosts the same backup facts as a human operator.

The project is not a hosted agent platform. It is a local Apple Notes backup control room with three truthful builder surfaces: a token-gated same-machine Local Web API, an advisory AI diagnosis sidecar, and a stdio-first, read-only-first MCP provider.

Three layers to remember

  • Category: local-first Apple Notes backup control room
  • AI/agent hook: AI Diagnose plus a read-only MCP surface
  • Result: humans and coding agents can inspect backup health, recent runs, and recovery clues from the same local state
Layer 1 CLI first

The deterministic control-room entrypoint is still the anchor of every builder story.

Layer 2 Web second

Use the Local Web API when your same-machine workflow already speaks browser or HTTP.

Layer 3 AI third

AI Diagnose is an explanation layer over operational state, not a second source of truth.

Layer 4 MCP fourth

MCP is the read-only-first agent substrate, not a remote platform or write-capable control plane.

Current best-fit ecosystem bindings

  • Primary: MCP-aware coding agents and local automation hosts
  • Natural secondary fit: Codex-, Claude Code-, and OpenClaw-style local workflows when the host can register or launch stdio MCP servers
  • Template-level guidance only: OpenCode and OpenHands, because this repository does not ship dedicated adapters, hosted runtimes, or repo-verified attach proof for them today
  • Not a truthful front-door binding: OpenClaw-style generic AI assistant positioning

What exists right now

  • `notesctl`: the canonical human entrypoint
  • Local Web API: a token-gated same-machine browser and HTTP lane backed by the same repo-owned command surface
  • AI Diagnose: an operator next-step assistant that explains local status, doctor, log-health, and recent-run evidence
  • MCP Provider: a read-only agent substrate that exposes local backup diagnostics and safe runtime summaries
  • State substrate: state.json, aggregate summaries, and local logs remain the underlying truth-bearing surfaces

Current builder entry points you can trust

  • Deterministic CLI reads: ./notesctl status --json, ./notesctl doctor --json, ./notesctl log-health --json, and ./notesctl aggregate --tail N
  • Token-gated local browser/API lane: ./notesctl web when your local workflow wants the Web console plus the same-machine JSON endpoints behind a local token gate
  • Advisory CLI layer: ./notesctl ai-diagnose --json when you want a calmer explanation around the same local facts
  • Agent substrate: ./notesctl mcp when your host can launch stdio MCP servers

Treat these as the truthful builder entry points today. This repository does not claim a hosted runtime, a public OpenAPI, or a generated SDK surface yet, even if your host is Codex or Claude Code.

Minimal stdio launch shape

If your local host can launch stdio MCP servers, the minimal launch contract is just the repo-owned entrypoint plus mcp. 

{
  "command": "/absolute/path/to/notesctl",
  "args": ["mcp"]
}

That is the narrow, truthful claim. Hosts differ in how they store this config, so this page documents the launch shape rather than pretending there is one universal agent setup file.

How the current layers map together

  • Human operator path: run, verify, doctor, status, dashboard, and the optional local Web console
  • Local browser/API path: web serves the token-gated control room plus same-machine JSON endpoints for status, doctor, log-health, recent-runs, and access
  • AI explanation path: ai-diagnose reads the deterministic state surfaces and summarizes them without becoming the system truth
  • Agent path: mcp exposes read-only tools and resources backed by the same local state

What this repository does not claim

  • No hosted agent runtime
  • No write-capable MCP surface in v1
  • No public OpenAPI or generated SDK surface today
  • No content-level AI, note QA, or semantic-search product positioning
